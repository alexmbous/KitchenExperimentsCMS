# Kitchen CMS — Learning Reference

A running reference for the KeystoneJS 6 project being built phase-by-phase.
Updated at the end of every phase.

---

## Project overview

**Stack**
- Keystone 6 (`@keystone-6/core` ^6, `@keystone-6/auth` ^8)
- TypeScript
- SQLite via Prisma (dev; trivially swappable to Postgres in production)
- Built-in local filesystem image storage under `public/images/`
- No additional libraries

**Core insight — one config, three surfaces**

`schema.ts` drives three surfaces simultaneously:
1. **Prisma schema + SQLite** — the data layer
2. **Admin UI** — auto-generated Next.js app at `http://localhost:3000/`
3. **GraphQL API** — auto-generated at `http://localhost:3000/api/graphql`

Adding one field in `schema.ts` creates a DB column, an Admin UI form input, and a GraphQL field with no other code changes. This is what Keystone *is*.

**Gotcha: `SESSION_SECRET` missing during `keystone postinstall`**

`keystone postinstall` spawns in Windows `cmd.exe`, which doesn't auto-load `.env`. Runtime (`keystone dev`) auto-loads it through its Next.js internals; postinstall doesn't. Fix applied to `auth.ts`:


`process.loadEnvFile()` is built into Node 20.12+ (we're on 22). The try/catch makes it a no-op when `.env` doesn't exist — production will pass the secret via real env vars.

## Phase 2 — Ingredient + Recipe↔Ingredient relationship ✅ complete (data layer)

**Goal:** Add an `Ingredient` list and wire a two-sided one-to-many relationship with `Recipe`.

**What this phase teaches:**
- Two-sided relationships via `ref: 'OtherList.fieldName'` — the `ref:` strings cross over, each side names *the other side's* field. That's how Keystone pairs the two fields into a single relation rather than two independent ones.
- How a single pair of declarations produces, with no additional code:
  - A foreign-key column in SQLite (on the `many: false` side)
  - Admin UI pickers on both sides
  - Resolvable GraphQL fields on both sides
  - An auto-generated `<field>Count` query (e.g. `Recipe.ingredientsCount`) — free for every `many: true` relation
  - Auto-generated plural mutations (`createIngredients`, `updateIngredients`, `deleteIngredients`) alongside the singular ones
  - Nested relationship input syntax: `{ connect: { id: "..." } }` to attach existing records; `{ create: {...} }` to create-and-attach in one call
- Cardinality: `many: true` vs `many: false`; the `many: false` side owns the foreign key. Both `many: true` would have produced a join table.
- Pragmatic field type choices: `amount` as `text` (not `integer`) so values like `"1/2"`, `"a pinch"`, `"to taste"` work.

**Files touched:** `schema.ts` only.

**Three valid modelling options — chose A:**
| Option | Shape | Verdict |
|---|---|---|
| **A. Two-sided one-to-many** | `Recipe.ingredients` many, `Ingredient.recipe` single | ✅ chosen |
| B. One-sided from Recipe only | Only `Recipe.ingredients` | ❌ can't traverse from an Ingredient to its Recipe |
| C. Many-to-many | `many: true` on both sides | ❌ would make a join table; breaks "each ingredient belongs to one recipe" |

**The two relationship fields in plain English:**
- `ingredients: relationship({ ref: 'Ingredient.recipe', many: true })` — "Recipe has many Ingredients; the other side of this relation is Ingredient.recipe"
- `recipe: relationship({ ref: 'Recipe.ingredients', many: false })` — "Ingredient has one Recipe; the other side is Recipe.ingredients"

**Sample mutation that exercises the two-sided relation:**

```graphql
mutation {
  createIngredients(data: [
    { name: "Flour", amount: "500", unit: "g", optional: false,
      recipe: { connect: { id: "<recipe-id>" } } },
    { name: "Salt", amount: "1", unit: "tsp", optional: false,
      recipe: { connect: { id: "<recipe-id>" } } }
  ]) {
    id name recipe { id title }
  }
}
```

**Verification completed (CLI):**
- Schema introspection confirms `Ingredient` type with exactly the 6 spec fields
- Schema introspection confirms `Recipe.ingredients: [Ingredient!]` *and* a bonus `Recipe.ingredientsCount: Int`
- `createIngredients` with nested `{ connect: { id } }` attached two ingredients to the existing recipe
- Forward query `{ recipes { ingredients { name } } }` returns the linked ingredients
- Reverse query `{ ingredients { recipe { title } } }` traverses the other direction — proves both sides are wired

**Still worth eyeballing in the browser when you have a minute:**
- Admin UI left nav now shows three lists (Users, Recipes, Ingredients)
- Opening an Ingredient shows a Recipe picker populated with existing recipes
- Opening your Recipe shows an Ingredients panel listing Flour and Salt — the auto-populated inverse side

**Gotcha: schema-change restarts**

When you add a new list or change cardinality, Keystone dev prints:
> `🔄 Your prisma schema has changed, please restart Keystone`

Hot-reload covers edits to existing fields but not structural changes. Stop with Ctrl+C and re-run `npm run dev`.

**Gotcha: Windows Prisma DLL lock**

On Windows, `keystone dev` loads `node_modules/.prisma/client/query_engine-windows.dll.node` and holds an OS-level file handle on it. When the next dev boot tries to regenerate the Prisma client, `rename()` on the `.tmp` file fails with `EPERM`:
> `Error: EPERM: operation not permitted, rename '…query_engine-windows.dll.node.tmp…' -> '…query_engine-windows.dll.node'`

This happens when a prior `keystone dev` process wasn't fully killed — port-based kills only target the child bound to :3000; the parent `keystone dev` process can survive and keep the DLL open. VSCode's TypeScript server can also load Prisma's client and lock the DLL.

Fix: kill *all* Node processes for this project, not just the port listener. PowerShell:

```powershell
Get-CimInstance Win32_Process -Filter "Name='node.exe'" |
  Where-Object { $_.CommandLine -like '*KitchenExperimentsCMS*' } |
  ForEach-Object { Stop-Process -Id $_.ProcessId -Force }
```

Then re-run `npm run dev`. If it still fails, restart VSCode.

## Phase 3 — CookingSession + relation to Recipe ✅ complete (data layer)

**Goal:** Add a `CookingSession` list to log each time a recipe is cooked, with outcome rating and iteration notes.

**What this phase actually taught:**
- The two-sided one-to-many pattern *again*, in a new domain — reinforcement, not a new concept. Same `ref: 'OtherList.field'` mechanic, same `many: true` on the parent and `many: false` on the child. This is the third time the pattern has appeared; should now be routine.
- **Bounded integer validation** via `integer({ validation: { min, max } })`. Boundary-violating inputs are rejected server-side before the DB write, returning a GraphQL error with extension `code: "KS_VALIDATION_FAILURE"` and a message like `"CookingSession.outcomeRating: value must be less than or equal to 5"`. Works identically through Admin UI and API.
- **Scope discipline under an "if practical" requirement** — `photos` was in the spec with an "if practical" qualifier. A proper multi-image solution needs a dedicated `Photo` list + a third relationship, which would roughly double Phase 3's code without teaching a new pattern. Skipped for now; can be added later as either a single `image()` field (one-liner) or a full `Photo` list (its own phase) when the use case is clear.

**Fields implemented:**
- `recipe` — `relationship({ ref: 'Recipe.sessions', many: false })`
- `cookedAt` — plain `timestamp()` (no default — sessions are often logged after the fact)
- `versionLabel` — plain `text()`
- `outcomeRating` — `integer({ validation: { min: 1, max: 5 } })`
- `changesMade` — `text({ ui: { displayMode: 'textarea' } })`
- `resultNotes` — `text({ ui: { displayMode: 'textarea' } })`
- `wouldMakeAgain` — `checkbox({ defaultValue: false })`
- `servingsMade` — plain `integer()`
- `photos` — **skipped this phase** (see rationale above)

**Recipe side:** added `sessions: relationship({ ref: 'CookingSession.recipe', many: true })` alongside the existing `ingredients` field.

**Files touched:** `schema.ts` only.

**Verification completed (CLI):**

## Phase 4 — Tag (m2m) + Lesson (o2m) ✅ complete (data layer)

**Goal:** Add a `Tag` list with a many-to-many relation to Recipe, and a `Lesson` list with a one-to-many relation to Recipe.

**What this phase actually taught:**
- **Many-to-many via `many: true` on both sides.** No extra syntax, no `@relation` annotation, no explicit join table — just flip the child side's `many: false` to `many: true` and Keystone asks Prisma to auto-create a hidden join table. The `ref:` strings still cross over in the same way.
- **Cardinality leaks through the GraphQL input shape.** `connect` takes an *array* when the target is `many: true` and a *single object* when it's `many: false`:
  - m2m: `recipes: { connect: [{ id }] }`
  - o2m: `recipe:  { connect: { id } }`
  One flag in `schema.ts` changes every surface consistently — DB, Admin UI picker, query shape, and mutation input.
- **The contrast is most visible in traversal.** A Tag's `recipes` is always a list; a Lesson's `recipe` is always a single object. Same `ref:` mechanic, different cardinality → different return type.
- **Scope-discipline over data-quality upgrades.** `Tag.name` is *not* unique-indexed, `Lesson.category` is `text()` not `select`, `Lesson.sortOrder` has no default. Each of these is a one-line upgrade when real-use feedback demands it; speculating on them now is polish.

**Fields implemented:**

*Tag*
- `name` — `text({ validation: { isRequired: true } })`
- `color` — plain `text()` (holds hex, CSS name, or any string — Keystone core has no color picker)
- `recipes` — `relationship({ ref: 'Recipe.tags', many: true })`

*Lesson*
- `title` — `text({ validation: { isRequired: true } })`
- `body` — `text({ ui: { displayMode: 'textarea' } })` (same pattern as Recipe.summary, instructions, etc.)
- `recipe` — `relationship({ ref: 'Recipe.lessons', many: false })`
- `category` — plain `text()` (no enum chosen — spec didn't provide one)
- `sortOrder` — plain `integer()` (no default; null-ordered rows until you set values)

**Recipe side:** added `tags` (m2m) and `lessons` (o2m) alongside the existing `ingredients` and `sessions`. Recipe now has four independent child relationships, all driven by the same `relationship(...)` mechanic.

**Files touched:** `schema.ts` only.

**Sample mutation contrast showing m2m vs o2m `connect` shape:**

```graphql
# m2m — connect takes an ARRAY
mutation {
  createTags(data: [
    { name: "Bread",   color: "#b5651d",
      recipes: { connect: [{ id: "<recipe-id>" }] } },
    { name: "Weekend", color: "teal",
      recipes: { connect: [{ id: "<recipe-id>" }] } }
  ]) { id name recipes { id title } }
}
```

```graphql
# o2m — connect takes a SINGLE OBJECT
mutation {
  createLesson(data: {
    title: "Salt restraint",
    body: "Dialling salt down a notch improved balance",
    category: "technique",
    sortOrder: 1,
    recipe: { connect: { id: "<recipe-id>" } }
  }) { id title recipe { id title } }
}
```

**Verification completed (CLI):**
- `Tag` type: 3 spec fields + `id` + auto-generated `recipesCount`
- `Lesson` type: 5 spec fields + `id`
- `Recipe` now exposes `tags`/`tagsCount` and `lessons`/`lessonsCount` alongside the existing 4 relation-derived fields
- Created two tags via `createTags` with nested array `connect`; both linked to the existing recipe
- Created one lesson via `createLesson` with nested object `connect`
- Forward query `{ recipes { tags { ... } lessons { ... } } }` returned both populated
- Reverse m2m query `{ tags { recipes { ... } } }` returned `recipes` as an array (the m2m tell)
- Reverse o2m query `{ lessons { recipe { ... } } }` returned `recipe` as a single object (the o2m tell)

**Still worth eyeballing in the browser:**
- Admin UI left nav now shows six lists (Users, Recipes, Ingredients, Cooking Sessions, Tags, Lessons)
- The Tag create/edit form shows a **multi-select** Recipe picker (m2m UI tell); the Lesson create/edit form shows a **single** Recipe picker (o2m UI tell)
- Recipe detail page now renders four independent inverse-side panels: Ingredients, Sessions, Tags, Lessons

## Phase 5 — Access control + seed script + reference queries ✅ complete

**Goal:** Replace `allowAll` with real rules, add seed data, and provide the three reference GraphQL queries.

**What this phase actually taught:**
- **The two halves of Keystone access.** *Operation* access answers "can you call this at all?" (e.g. `createRecipe`). *Filter* access answers "which rows count?" (e.g. `status === 'published'`). They compose: operation runs first, filter narrows the result set.
- **Filter access on *child* lists is the leak-prevention mechanism.** Putting a filter on `Recipe` alone isn't enough — `{ ingredients { recipe { ... } } }` would expose draft content via relationship traversal. Each child list (Ingredient, CookingSession, Lesson) filters by its parent's `status`, using the same filter mechanism, one level of indirection: `{ recipe: { status: { equals: 'published' } } }`.
- **Filter propagates through relation resolution automatically.** You don't filter `tag.recipes` explicitly; because `Recipe`'s own list filter drops drafts for anonymous users, any traversal that resolves through the Recipe list gets the same filter for free. That's why `Tag` itself needed no filter config.
- **`operation.query: false` behaves like a filter, not a throw.** Keystone's convention: blocking a list query returns an empty array, not an `ACCESS_DENIED` error. Blocking a *mutation* returns the error. Noteworthy — don't expect a visible error from an anon `users` query.
- **`context.sudo()` is the escape hatch for server code.** `getContext(config, PrismaModule).sudo()` gives you the same GraphQL-style context as HTTP resolvers, but with access checks bypassed. It's how `seed.ts` can create drafts and connect items across lists without authenticating first.

**Access rules implemented:**

| List | Operation `query` | Operation `create/update/delete` | Filter `query` |
|---|---|---|---|
| `User` | `isSignedIn` | `isSignedIn` | (none) |
| `Recipe` | open | `isSignedIn` | anon → `status: published` |
| `Ingredient` | open | `isSignedIn` | anon → `recipe.status: published` |
| `CookingSession` | open | `isSignedIn` | anon → `recipe.status: published` |
| `Lesson` | open | `isSignedIn` | anon → `recipe.status: published` |
| `Tag` | open | `isSignedIn` | (none — leak prevented by Recipe's filter on reverse traversal) |

**Files touched:** `schema.ts` (swap `allowAll` for per-list access configs), `package.json` (add `seed` script + `tsx` dev dep).
**Files added:** `access.ts`, `seed.ts`, `queries.md`.

**Seed data:** 3 recipes (Sourdough + Weeknight Pasta *published*, Experimental Miso Broth *draft*), 4 tags, 10 ingredients, 3 cooking sessions, 2 lessons. The draft recipe is intentional — it's what makes the anonymous access verification meaningful.

**Library added:** `tsx` (dev dep) — explained in-line in `package.json`; Keystone 6 has no built-in seed command and the standard pattern is `tsx seed.ts`. Only library added across all five phases.

**Verification completed (CLI):**
- Anonymous `recipes` query returns 3 published (draft hidden) ✅
- Anonymous fetch of draft slug returns `null` ✅
- Anonymous query for ingredient/lesson of draft recipe returns `[]` (no leak via traversal) ✅
- Anonymous `createTag` mutation rejected with `KS_ACCESS_DENIED` ✅
- Anonymous `users` query returns `[]` (no email leak) ✅
- Anonymous traversal `tag(Comfort).recipes` returns `[]` (only draft attached; Recipe filter kicks in) ✅
- All three reference queries (`PublishedRecipes`, `RecipeBySlug`, `RecipeDeep`) return expected shapes ✅

**Still worth eyeballing in the browser (authenticated):**
- You should still be able to see the draft Miso Broth in the Admin UI
- Create/update/delete still works across all lists
- The `queries.md` queries can also be run in the Apollo sandbox

| Many-to-many relationships | Phase 4 | Both sides `many: true` → Keystone auto-creates a hidden join table in Prisma; neither side owns a foreign key |
| Cardinality shapes `connect` input | Phase 4 | m2m takes `connect: [{ id }]` (array); o2m takes `connect: { id }` (single); mirrors return-shape of the field |
| Operation vs filter access | Phase 5 | `access.operation.*` gates whether you can call an op; `access.filter.*` narrows which rows count. Both compose per list. |
| Filter access on child lists plugs leaks | Phase 5 | Parent filter alone doesn't protect traversal; each child filters by parent status to close the `{ ingredients { recipe { ... } } }` back-door. |
| `operation.query: false` silently returns `[]` | Phase 5 | Keystone convention — blocked list queries filter to empty, not throw. Blocked mutations throw `KS_ACCESS_DENIED`. |
| `context.sudo()` for server-side code | Phase 5 | `getContext(config, PrismaModule).sudo()` gives bypass-all-access context for seeds, admin scripts, background jobs. |
| Server components as a server-side HTTP client | Phase 6 | Next.js server components run on the Next server, not the browser — they call Keystone's GraphQL endpoint as an anonymous HTTP client. No CORS, no session cookies to forward. |
| Access control propagates to the frontend for free | Phase 6 | Because the backend filter runs on every query, drafts simply don't exist in the data a server component sees. The frontend needs zero draft-handling logic. |
| `notFound()` as the access-denied pattern | Phase 6 | When GraphQL returns `null` for a slug (because access filter hid it), Next's `notFound()` renders a 404 — clean way to surface "not publicly accessible" without leaking existence. |
| `server.extendExpressApp` + `server.cors` | Phase 7 | Keystone's two first-class doors for Express middleware and CORS config; no forking, no wrapping — pass the config, Keystone plugs it in. |
| Scoped middleware via path prefix | Phase 7 | `app.use('/api/graphql', limiter)` mounts middleware on just that path; Admin UI, images, and other routes are unaffected. |
| Cookie-presence as auth heuristic | Phase 7 | Rate limiter's `skip` checks `req.headers.cookie?.includes('keystonejs-session=')` — no validation, just presence; forgery only skips the limit for the forger, real auth still enforced server-side. |
| Env-driven prod vs dev config | Phase 7 | Single codebase; `process.env.NODE_ENV === 'production'` flips `secure: true` on cookies; `secure: false` in dev is required (localhost HTTP can't set `Secure` cookies). |
| Roles without a role system | Phase 8 | One `role` select field + `isAdmin` helper is enough for admin/editor tiers. `sessionData: 'id name email role'` makes `session.data.role` available in every access function. |
| Time-based filter access | Phase 8 | `publishedAt: { lte: new Date().toISOString() }` inside a filter function re-evaluates on every query — scheduled publishing with zero cron jobs or background workers. |
| Null `publishedAt` is hidden by `lte` | Phase 8 | `null` can't satisfy `lte`, so recipes without a publish date are hidden from anonymous traversal by the same filter — no extra null-check needed. |
| Next `searchParams` for URL-backed state | Phase 8 | Server component reads `searchParams` as a plain prop, builds a GraphQL variable, re-renders. Refresh, share, back button all work. No `"use client"`, no `useState`, no router hooks. |
| `authenticateUserWithPassword` mutation | Phase 8 | Keystone's built-in login mutation returns a union (`UserAuthenticationWithPasswordSuccess` \| `Failure`) and sets the session cookie via `Set-Cookie` — save with `curl -c`, send with `curl -b`. |
| Keystone hydrates `sessionData` per-request | Phase 8 | Even with stateless session cookies, the fields in `sessionData` are re-queried from the DB on each request — so role changes take effect immediately, no re-login needed. |

| Issue | Phase | Fix |
|---|---|---|
| `SESSION_SECRET must be set in .env` during `npm install` | 1 | `process.loadEnvFile()` at top of `auth.ts` (Node 20.12+ built-in) |
| `🔄 Your prisma schema has changed, please restart Keystone` | 2 | Add-list / cardinality-change requires dev server restart (Ctrl+C, `npm run dev`) |

| `Error: ENOENT: no such file or directory, stat '…@keystone-6/core/scripts/cli/static/dev-loading.html'` on dev boot | 3 | **Cosmetic — ignore.** Keystone tries to serve an internal loading splash from a file that isn't shipped in some builds. | Adding a `select` field with `defaultValue` applies the default to all existing rows, not `null` | 8 | Prisma fills existing rows with the default on column add — so a "backfill only where role is null" script misses every row. Fix: target the specific bootstrap user by email (or some other identifying criterion) and promote explicitly, not via a null check. |

## Phase 6 — Tiny public frontend ✅ complete

**Goal:** Ship a minimal public-facing site that consumes the Keystone GraphQL API as an anonymous user. Two routes: published recipes list, recipe detail by slug.

**What this phase actually taught:**
- **Server components are a separate HTTP client, not browser code.** Next.js's server process calls `http://localhost:3000/api/graphql` the same way `curl` does — identically anonymous, identically subject to Phase 5's access control. No cookies are attached unless you explicitly forward them. This is why there's no CORS issue and no need for client-side draft filtering.
- **Phase 5's access rules enforce through the frontend automatically.** The homepage queries `recipes` → the backend filter hides drafts → the frontend simply doesn't see them. No frontend code knows drafts exist. This is the payoff of having put the filter on the right layer.
- **`notFound()` is the clean pattern for access-blocked resources.** When a user hits `/recipes/experimental-miso-broth`, GraphQL returns `null` (access filter dropped it), the page calls `notFound()`, Next renders a 404. Doesn't leak the existence of a draft slug — the backend does the deciding.
- **"Tiny" is a scope discipline.** Resisting the urge to add Tailwind, a GraphQL client, a component folder, a shared fetch helper — each of which is a valid choice — is what keeps the frontend genuinely minimal. Eight files total, two of them boilerplate that Next expects.

**Files added (in `web/`):** `package.json`, `tsconfig.json`, `next.config.js`, `next-env.d.ts`, `app/globals.css`, `app/layout.tsx`, `app/page.tsx`, `app/recipes/[slug]/page.tsx`. Zero changes to the Keystone backend.

**Key implementation picks:**
- **Next.js 14.2 with App Router** — latest stable 14 line; `params` is a plain object (not a Promise like in 15), which keeps page files shorter.
- **Server components only** — no `"use client"` anywhere. Every data fetch happens server-side. Sidesteps CORS entirely for this phase.
- **Native `fetch()`** — no Apollo, no `graphql-request`, no `urql`. POST to the GraphQL endpoint with a JSON body containing `query` and `variables`. ~15 lines per page.
- **`cache: 'no-store'`** on both fetches — seed data changes in dev; always-fresh responses are easier to reason about than revalidation windows. Can flip to revalidation in a later phase.
- **Plain `<a>` tags, not `next/link`** — one less import per file; prefetching isn't worth the complexity for two pages.
- **Explicit TypeScript types per page** co-located with each query. ~12 lines each, pay for themselves at the render site.
- **Inline fetch logic in each page**, not a shared `lib/api.ts` helper — two pages with 6 lines of fetch each beats adding a folder structure.

**Verification completed:**
- Next boots on `:3001` in ~3s
- Homepage returns 3 published recipes (Sourdough, Weeknight Pasta, test title); draft Miso Broth absent
- `/recipes/sourdough` renders ingredients (4), cooking log (1 session), lessons (1), tags (2), instructions
- `/recipes/experimental-miso-broth` returns HTTP 404 from `notFound()`; draft title does not appear in 404 body
- `web/package.json` contains zero GraphQL client libraries — only Next, React, and types

**Dual-process dev workflow is now the norm:** Keystone on `:3000` in one terminal, Next on `:3001` in another. Both must be running for the frontend to show live data.

## Phase 7 — Backend hardening ✅ complete

**Goal:** Move the backend from "works in dev" to "plausibly deployable" by adding CORS, rate limiting, and env-aware session security — without changing the product model.

**What this phase actually taught:**
- **Keystone has first-class hooks for Express middleware.** Two doors: `server.cors` (Keystone wires it into the Express app it already runs) and `server.extendExpressApp(app => ...)` for arbitrary middleware. You don't fork Keystone or wrap its server — you pass config and Keystone plugs it in.
- **Rate limiting is an Express concern, not a GraphQL one.** Mount `express-rate-limit` scoped to the `/api/graphql` path and you're done. The same technique works for any other route Keystone exposes. No GraphQL-aware limiter needed.
- **Cookie presence as an auth heuristic.** The rate limiter's `skip` function checks for a `keystonejs-session=` cookie in the request headers — no validation, just presence. Good enough because a well-formed attacker can't forge a *signed* cookie anyway, so forgery only skips the limit for themselves, not for real users.
- **Prod differs from dev in a few specific cookie flags, not a separate codebase.** Flipping `secure: isProduction` on `statelessSessions` is enough for "I could deploy this." In dev `secure: false` is essential — browsers refuse to set `Secure` cookies over plain `http://localhost`.
- **CORS is a browser-only concern; server-to-server calls (like Phase 6's server components) don't trigger it.** Configuring CORS now is deployment preparation — so a future client-side fetch or a separately-hosted frontend works without code changes.

**Files touched:** `keystone.ts` (add `server.cors` + `server.extendExpressApp` with rate limiter), `auth.ts` (add `isProduction` + `secure`/`sameSite` on session), `package.json` (add `express-rate-limit`).

**Library added:** `express-rate-limit` (dep) — standard Express middleware, ~4 KB, does exactly what's needed. Total libraries added across all phases: 2 (`tsx` for seeding in Phase 5; `express-rate-limit` for hardening in Phase 7).

**Environment variables:**

| Variable | Default | Purpose |
|---|---|---|
| `SESSION_SECRET` | required | Signs session cookie (existing since Phase 1) |
| `CORS_ORIGIN` | `http://localhost:3001` | Comma-separated allowed frontend origins |
| `NODE_ENV` | `undefined` | When `production`, session cookie gets `Secure` flag |

**Verification completed:**
- **Phase 6 frontend** still renders identically — server-to-server calls don't touch CORS
- **CORS allowed origin** (`http://localhost:3001`) gets `Access-Control-Allow-Origin` + `Access-Control-Allow-Credentials: true` on preflight
- **CORS bad origin** (`http://evil.example.com`) gets a 204 but no `Access-Control-Allow-Origin` for that origin — browser will block
- **Anonymous rate limit** — 60 × 200 then immediate 429s on the 61st request and beyond, exactly matching the 60-per-minute window
- **Authenticated requests bypass** — 100 consecutive requests with a `keystonejs-session=` cookie all returned 200
- **RateLimit-* headers** emitted per draft-7 spec: `RateLimit-Policy: 60;w=60`, `RateLimit: limit=60, remaining=<n>, reset=<s>`

**Not verified at runtime (code-inspection only):** the `secure: true` cookie flag in `NODE_ENV=production` mode. Testing requires building + running prod mode and inspecting the `Set-Cookie` header, which is disruptive for a learning setup. The diff in `auth.ts` is the whole behavior.

**Deliberately out of scope (flagged by the user):**
- Audit fields (`createdBy`, `updatedAt`) — accountability is product-shaping, not hardening. Deferred.
- Role systems, password reset, email verification, 2FA, security headers (Helmet), etc.

## Phase 8 — Product shaping (roles + scheduled publishing + search) ✅ complete

**Goal:** Shape the project into a real cooking content platform — admin/editor roles, time-based publishing, and public search/filter — without bloating scope.

**What this phase actually taught:**
- **Roles without a role system.** One `role` select field + one `isAdmin` helper is enough to express two tiers of authority. Phase 5's access pattern composes cleanly: `adminOnly` for the User list, `contentWrites` for every other list (create/update require sign-in; delete requires admin). No RBAC matrix, no permission tables.
- **"Scheduled publishing" is a filter, not a job.** Adding `publishedAt: { lte: new Date().toISOString() }` to the anonymous filter means the comparison runs on every query. A recipe with `publishedAt` in the future is hidden today; it becomes visible the moment time catches up — no cron, no background worker, no `setInterval`.
- **Null `publishedAt` is hidden by the same mechanism.** `null` can't satisfy `lte`, so recipes without any publish date are also hidden from anonymous traversal. This is the right default for a content platform: you must deliberately publish.
- **Server components + `searchParams` is a complete URL-backed state pattern.** The homepage stays a server component; `searchParams` is just another prop alongside `params`. Build a GraphQL `where` from it, render, done. Refresh/share/back-button all work because state is in the URL, not in React. Zero client-side JavaScript added.
- **Keystone hydrates `sessionData` per-request.** Even with stateless session cookies, the fields named in `sessionData: 'id name email role'` are re-queried from the DB on every request. Means a role change takes effect immediately — no sign-out/sign-in dance.

**Access rules (final shape):**

| List | `query` | `create/update` | `delete` | Filter on anonymous `query` |
|---|---|---|---|---|
| `User` | `isAdmin` | `isAdmin` | `isAdmin` | (none; op-level blocks anon already) |
| `Recipe` | open | `isSignedIn` | `isAdmin` | `status: published AND publishedAt <= now` |
| `Ingredient` | open | `isSignedIn` | `isAdmin` | `recipe.status: published AND recipe.publishedAt <= now` |
| `CookingSession` | open | `isSignedIn` | `isAdmin` | same as Ingredient |
| `Lesson` | open | `isSignedIn` | `isAdmin` | same as Ingredient |
| `Tag` | open | `isSignedIn` | `isAdmin` | (none; leak prevented by Recipe filter on reverse traversal) |

**Files touched:** `access.ts` (rename + add `isAdmin` + add time clause to both filters), `schema.ts` (add `role` to User; rename imports; swap access helpers on all lists), `auth.ts` (`sessionData` adds role; `initFirstItem.itemData` pins admin), `seed.ts` (role backfill + test editor user + future-dated recipe), `queries.md` (add `$now` to PublishedRecipes), `web/app/page.tsx` (searchParams-driven filter UI), `web/app/globals.css` (~12 lines for search form + tag chips).

## Things I'm deliberately *not* doing

Final list after Phase 8. Kept out of scope on purpose; each has a clear path to being added later when the need is real:

- **Audit fields** (`createdBy`, `updatedAt`) — deferred from Phase 7 and still out of scope after Phase 8. Accountability is a distinct feature; the `session.data.id` is already available via `defaultValue: ({ context }) => context.session?.data?.id` when needed.
- **Multi-image `photos` on `CookingSession`** — deferred after Phase 3; two clean add-later paths: (a) single `image()` field on `CookingSession` (one-liner, reuses `local_images` storage) or (b) dedicated `Photo` list + two-sided relation (own mini-phase).
- **Tag-name uniqueness** — `Tag.name` is not uniquely indexed; duplicate names are currently allowed. One-line add: `isIndexed: 'unique'`.
- **`Lesson.category` as a fixed `select`** — currently `text()`. Promote to `select({ type: 'enum', options: [...] })` once the real vocabulary is known.
- **`Lesson.sortOrder` default** — currently unset rows are `null`, which sorts arbitrarily. Add `defaultValue: 0` when null-ordering becomes annoying in real use.
- **More than two roles** — current design supports `admin` and `editor` only; any more needs a small refactor of the access helpers.
- **Tag-name slug in URLs** — currently the public homepage uses `?tag=<name>` verbatim. Tag name collisions would break this; can be addressed by adding `isIndexed: 'unique'` to `Tag.name` or by switching to a slug field.
- Rich-text / document field (not required by spec)
- Email verification, password reset, magic links, 2FA (not required)
- API keys / machine-to-machine auth (not required)
- Security headers beyond CORS (Helmet etc.) (not required)
- Tests (not required by spec — real projects would add Vitest + a GraphQL test client here)
- Production deployment, Postgres swap, cloud image storage (all are config-only changes Keystone supports)
